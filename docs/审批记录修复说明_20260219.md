# 审批记录修复说明

## 问题描述

**问题现象**：
- 高级权限用户（如 CFO、财务经理）处理完全部待审单据后
- 处理完的单据不会出现在"我已处理"标签页中
- 导致用户无法查看自己的审批历史记录

**影响范围**：
- 所有权限的审批人员
- 特别是处理低风险和中风险单据的高级权限用户

## 问题原因

### 原有逻辑缺陷

在 `routes/approvals.py` 文件中，处理审批动作时的逻辑存在问题：

**原代码（有问题）**：
```python
# 低/中风险单据通过
else:
    updates["approval_stage"] = "DONE"
    updates["approval_status"] = "APPROVED"
    updates["status"] = "APPROVED"
    updates["queue_owner_id"] = ""
    # 问题：只有当 first_approver_id 为空时才记录
    if not _safe_text(row.get("first_approver_id")):
        updates["first_approver_id"] = actor
    if not _safe_text(row.get("first_approved_at")):
        updates["first_approved_at"] = now_text

# 退回/驳回
elif action in {"REJECT", "RETURN"}:
    # ...
    # 问题：只有当 first_approver_id 为空时才记录
    if not _safe_text(row.get("first_approver_id")):
        updates["first_approver_id"] = actor
    if not _safe_text(row.get("first_approved_at")):
        updates["first_approved_at"] = now_text
```

**问题分析**：
1. 代码检查 `first_approver_id` 是否为空
2. 只有为空时才记录当前审批人
3. 如果字段已有值（即使是其他人），就不会更新
4. 导致实际处理人没有被记录

### 前端筛选逻辑

前端 `static/js/approval_center.js` 中的筛选逻辑：

```javascript
function processedByMe(el) {
  return (
    identities.has(text(el.getAttribute("data-first"))) ||
    identities.has(text(el.getAttribute("data-second")))
  );
}
```

前端逻辑是正确的，它会检查：
- `first_approver_id`（一级审批人）
- `second_approver_id`（二级审批人）

只要当前用户的身份匹配其中任何一个，就会显示在"我已处理"中。

**但是**，由于后端没有正确记录审批人，前端无法找到匹配的记录。

## 修复方案

### 修改内容

修改 `routes/approvals.py` 文件，确保每次审批动作都记录当前操作人：

**修复后的代码**：
```python
# 低/中风险单据通过
else:
    # 低/中风险单据：一次性通过
    updates["approval_stage"] = "DONE"
    updates["approval_status"] = "APPROVED"
    updates["status"] = "APPROVED"
    updates["queue_owner_id"] = ""
    # 始终记录当前审批人
    updates["first_approver_id"] = actor
    updates["first_approved_at"] = now_text

# 退回/驳回
elif action in {"REJECT", "RETURN"}:
    if not comment:
        return jsonify({"ok": False, "msg": "comment is required for reject/return"}), 400
    target_status = "REJECTED" if action == "REJECT" else "RETURNED"
    updates["approval_stage"] = "DONE"
    updates["approval_status"] = target_status
    updates["status"] = target_status
    updates["queue_owner_id"] = ""
    # 始终记录当前审批人
    updates["first_approver_id"] = actor
    updates["first_approved_at"] = now_text
```

### 修复要点

1. **移除条件判断**：不再检查 `first_approver_id` 是否为空
2. **始终记录**：每次审批动作都记录当前操作人
3. **覆盖更新**：如果字段已有值，会被当前操作人覆盖
4. **保持一致**：通过、退回、驳回三种动作都使用相同的记录逻辑

### 审批流程说明

修复后的完整审批流程：

#### 1. 低风险/中风险单据（一次性审批）
```
待审 → [任何有权限的人审批] → 完成
记录：first_approver_id = 当前审批人
结果：该审批人可以在"我已处理"中看到此单据
```

#### 2. 高风险单据（两级审批）
```
待审 → [一级审批] → 待二级复核 → [二级审批] → 完成
记录：
  - first_approver_id = 一级审批人
  - second_approver_id = 二级审批人
结果：两位审批人都可以在"我已处理"中看到此单据
```

#### 3. 退回/驳回
```
待审 → [审批人退回/驳回] → 完成
记录：first_approver_id = 当前审批人
结果：该审批人可以在"我已处理"中看到此单据
```

## 影响评估

### 正面影响

1. **功能完整性**：修复了"我已处理"功能缺陷
2. **用户体验**：用户可以查看自己的审批历史
3. **审计追溯**：完整记录每个审批人的操作
4. **权限公平**：所有权限级别的用户都能正常使用

### 潜在影响

1. **历史数据**：
   - 修复前处理的单据可能没有正确记录审批人
   - 这些历史记录不会自动修复
   - 只影响修复后的新审批记录

2. **数据覆盖**：
   - 如果同一单据被多次处理（理论上不应该发生）
   - 会记录最后一次处理人
   - 这是预期行为，因为单据状态已经是最终状态
