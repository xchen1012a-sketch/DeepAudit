# admin01 权限问题修复指南

## 问题描述

部署到 GitHub 并上线后，使用 admin01 账号登录显示"系统暂时不可用，服务器内部错误"，提示无权限访问。

## 问题原因

根据诊断结果，本地数据库中 admin01 账号配置完全正常，但部署到服务器后出现权限问题。可能的原因：

1. **服务器数据库文件与本地不同步** - 最可能的原因
2. **数据库文件路径配置错误** - 服务器读取了错误的数据库文件
3. **数据库文件权限问题** - 服务器无法写入数据库
4. **环境变量配置错误** - DATABASE_URL 或 DB_PATH 配置不正确

## 修复方案

### 方案 1：在服务器上运行修复脚本（推荐）

1. **上传修复脚本到服务器**
   ```bash
   # 确保 fix_admin01_permissions.py 已上传到项目根目录
   ```

2. **在服务器上运行修复脚本**
   ```bash
   cd /path/to/DeepAudit2
   python fix_admin01_permissions.py
   ```

3. **重启应用**
   ```bash
   # 如果使用 systemd
   sudo systemctl restart deepaudit
   
   # 如果使用 gunicorn
   pkill -HUP gunicorn
   
   # 如果使用 supervisor
   sudo supervisorctl restart deepaudit
   ```

4. **清除浏览器缓存并重新登录**

### 方案 2：上传本地数据库到服务器

如果方案 1 不起作用，可以直接上传本地的数据库文件：

1. **备份服务器上的数据库**
   ```bash
   cd /path/to/DeepAudit2
   cp database.db database.db.backup.$(date +%Y%m%d_%H%M%S)
   ```

2. **上传本地数据库到服务器**
   ```bash
   # 在本地执行
   scp C:\Users\画桦\Downloads\DeepAudit2-main\database.db user@server:/path/to/DeepAudit2/
   ```

3. **设置正确的文件权限**
   ```bash
   # 在服务器上执行
   cd /path/to/DeepAudit2
   chmod 644 database.db
   chown www-data:www-data database.db  # 根据实际运行用户调整
   ```

4. **重启应用**

### 方案 3：检查数据库路径配置

1. **检查环境变量**
   ```bash
   # 在服务器上执行
   cd /path/to/DeepAudit2
   python -c "import os; from pathlib import Path; print('DB_PATH:', os.getenv('DB_PATH', 'not set')); print('DATABASE_URL:', os.getenv('DATABASE_URL', 'not set'))"
   ```

2. **检查实际使用的数据库文件**
   ```bash
   python -c "from utils.db import DB_PATH; print('实际数据库路径:', DB_PATH); import os; print('文件存在:', os.path.exists(DB_PATH))"
   ```

3. **如果路径不正确，设置环境变量**
   ```bash
   # 方法 1: 在启动脚本中设置
   export DATABASE_URL="sqlite:///database.db"
   python app.py
   
   # 方法 2: 创建 .env 文件
   echo "DATABASE_URL=sqlite:///database.db" > .env
   ```

### 方案 4：使用 SQL 直接修复（高级）

如果无法运行 Python 脚本，可以直接执行 SQL：

```bash
# 在服务器上执行
cd /path/to/DeepAudit2
sqlite3 database.db << 'EOF'

-- 4. 创建所有必需的权限
INSERT OR IGNORE INTO permissions (permission_key) VALUES

-- 5. 为系统管理员角色分配所有权限
INSERT OR IGNORE INTO role_permissions (role_id, permission_id)

-- 6. 验证结果
SELECT '用户信息:' as info;
SELECT id, username, employee_name, role, status FROM users WHERE username = 'admin01';

SELECT '角色关联:' as info;
SELECT ur.user_id, ur.role_id, r.role_name 
FROM user_roles ur 
JOIN roles r ON r.id = ur.role_id 
WHERE ur.user_id = (SELECT id FROM users WHERE username = 'admin01');

SELECT '权限数量:' as info;
SELECT COUNT(*) as permission_count
FROM role_permissions rp
JOIN user_roles ur ON ur.role_id = rp.role_id
WHERE ur.user_id = (SELECT id FROM users WHERE username = 'admin01');
EOF
```

## 常见问题

### Q1: 修复脚本运行失败，提示数据库锁定
**原因**: 应用正在运行，数据库被锁定

**解决**:
```bash
# 先停止应用
sudo systemctl stop deepaudit

# 运行修复脚本
python fix_admin01_permissions.py

# 重启应用
sudo systemctl start deepaudit
```

### Q2: 修复后仍然无权限
**排查步骤**:

1. 检查是否读取了正确的数据库文件
   ```bash
   python -c "from utils.db import DB_PATH; print(DB_PATH)"
   ```

2. 检查数据库文件权限
   ```bash
   ls -l database.db
   # 应该可以被应用运行用户读写
   ```

3. 检查代码版本是否正确
   ```bash
   git log -1 --oneline
   # 确认是最新的代码
   ```

4. 检查 utils/security.py 中的 SYSTEM_ADMIN_USERNAMES
   ```bash
   grep -A 2 "SYSTEM_ADMIN_USERNAMES" utils/security.py
   # 应该包含 "admin01"
   ```

### Q3: 本地正常，服务器不正常
**可能原因**:

1. **代码版本不一致** - 确保服务器代码是最新的
   ```bash
   git pull origin main
   ```

2. **Python 版本不同** - 检查 Python 版本
   ```bash
   python --version
   # 应该是 3.8 或更高
   ```

3. **依赖包版本不同** - 重新安装依赖
   ```bash
   pip install -r requirements.txt --upgrade
   ```

### Q4: 如何强制所有用户重新登录
```bash
# 如果 session 存储在文件中
rm -rf flask_session/*

# 如果 session 存储在 Redis 中
redis-cli FLUSHDB

# 或者直接重启应用（如果 session 在内存中）
sudo systemctl restart deepaudit
```

## 预防措施

为避免将来再次出现此问题：

1. **使用版本控制管理数据库迁移**
   - 不要直接提交 database.db 到 Git
   - 使用数据库迁移脚本（如 Alembic）

2. **在部署脚本中包含数据库初始化**
   ```bash
   # deploy.sh
   python scripts/init_db_keep_admin01.py --yes
   python fix_admin01_permissions.py
   ```

3. **设置健康检查**
   - 定期检查 admin01 账号权限
   - 监控登录失败日志

4. **备份数据库**
   ```bash
   # 每天自动备份
   0 2 * * * cd /path/to/DeepAudit2 && cp database.db backups/database.db.$(date +\%Y\%m\%d)
   ```
